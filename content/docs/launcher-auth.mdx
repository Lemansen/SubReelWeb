# Авторизация лаунчера

Сайт и лаунчер уже могут работать через один аккаунт. База авторизации, сессии сайта и отдельный `launcher token` уже подготовлены в текущем проекте.

## Какой flow сейчас готов

1. Пользователь создаёт аккаунт на сайте или входит в существующий.
2. В кабинете открывает блок `Launcher Access`.
3. Копирует `launcher token` или обновляет его.
4. Лаунчер может авторизоваться через API и получить профиль пользователя.

## Что уже есть в API

### Вход лаунчера

`POST /api/launcher/auth`

Отправь JSON:

```json
{
  "identifier": "login-or-email",
  "password": "secret-password"
}
```

Ответ:

```json
{
  "ok": true,
  "launcher": {
    "token": "launcher-token",
    "user": {
      "id": "user-id",
      "login": "lemansen",
      "nickname": "Lemansen",
      "role": "player",
      "launcherToken": "launcher-token",
      "microsoftConnected": false,
      "lastLoginAt": "2026-03-24T10:00:00.000Z"
    }
  }
}
```

### Проверка сессии лаунчера

`GET /api/launcher/session`

Заголовок:

```txt
Authorization: Bearer launcher-token
```

### Обновление launcher token на сайте

`POST /api/account/launcher-token`

Этот маршрут работает из кабинета сайта и требует обычную cookie-сессию пользователя.

## Как лучше использовать это в клиенте

- Храни `launcher token` отдельно от обычной веб-сессии.
- При старте лаунчера сначала проверяй `GET /api/launcher/session`.
- Если токен стал невалидным, показывай форму входа заново.
- Не смешивай `launcher token` и браузерную cookie-сессию: это разные контуры доступа.

## Что логично делать следующим шагом

- добавить Microsoft-вход к той же модели аккаунта
- отдавать лаунчеру больше данных профиля игрока
- связать роли, whitelist и доступ к серверу с одним аккаунтом
